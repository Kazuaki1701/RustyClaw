## 1. 開発環境の役割分担

安全に稼働し続けるエージェントシステムを維持するため、開発は以下の3層構造で進めます。

| 環境 | 役割 | 遵守事項 |
| :--- | :--- | :--- |
| **Local** (開発機) | 自由な作業・実験・事前検証の場 | コミット前にローカルでの clippy と test の通過を確認する |
| **Remote** (GitHub) | 品質保証（CI）・正史の保管庫 | GitHub Actions による自動検証（Clippy & Test）がグリーンであることをマージの絶対条件とする |
| **rp1** (本番機) | エージェントの24時間常時稼働環境 | CIで保証された `main` ブランチの最新コードのみをデプロイする |

## 2. Git ブランチ & コミットルール

### 2-1. `main` ブランチへの直接コミット禁止
* 履歴の混乱を防ぎ、CIでの検証を強制するため、`main` ブランチへ直接コミットしたり直接履歴を変更することは厳禁です。

### 2-2. トピックブランチの作成
* すべての変更はトピックブランチを切り、そこで作業を行います。
* ブランチ名のプレフィックス規則:
  * 新機能開発: `feat/xxx`
  * バグ修正: `fix/xxx`
  * ドキュメント更新: `docs/xxx`
  * リファクタリング: `refactor/xxx`

### 2-3. コミットメッセージの記述ルール
* 変更内容を追跡しやすくするため、メッセージは以下の形式を徹底します。
* **フォーマット**: `<type>(<scope>): Phase <No> (<Issue番号>) <変更内容の要約>`
  * 例: `fix(agent): Phase 28b-4 (#42) Groqのトークンオーバーフロー対策`
  * ※特定の課題番号が無い場合は `docs(rules): GitHub Actions 連携と新規課題追加フローの明文化` のように記述します。

## 3. プルリクエスト（PR）と CI 検証

### 3-1. ローカルでの事前検証
* コミット・プッシュする前に、開発者はローカルで以下の検証を行い、警告が 0 件であることを確認します。
  ```bash
  cargo clippy --all-targets --all-features -- -D warnings
  cargo test --all-features --workspace
  ```

### 3-2. PRの作成と CI の通過義務
* 機能実装が完了したら PR（Pull Request）を作成し、GitHub Actions による CI（自動ビルド、テスト、Clippy）を実行します。
* **マージの絶対条件**:
  * GitHub Actions 上のジョブ結果が **`✓` (Success)** になるまで、絶対にマージを行ってはいけません。
  * ローカルでテストが成功しても、CI環境（UTCなど）で失敗する場合があるため、必ず CI のグリーンを確認します。

## 4. マージ手順とブランチ削除

### 4-1. `--no-ff` マージの強制
* トピックの境界と履歴の整合性を視覚的に明確にするため、`main` への統合は必ず `--no-ff` (No Fast-Forward) オプションを使用します。
* **コマンド例**:
  ```bash
  git checkout main
  git merge --no-ff <branch_name> -m "Merge branch '<branch_name>' into main"
  git push origin main
  ```

### 4-2. マージ後のクリーンアップ
* `main` へのマージ完了後、不要となったトピックブランチはローカルおよびリモート（`origin`）から直ちに削除します。
* **コマンド例**:
  ```bash
  git branch -d <branch_name>
  git push origin --delete <branch_name>
  ```

## 5. 新規課題・案件の追加フロー

### 5-1. 案件（新機能・Phase）の追加
1. **GitHub Issue 起票**: GitHub 上で Issue を作成し、`Phase XX-Y` の管理番号と Issue 番号（例: `#15`）を確定させます。
2. **タスク登録**: `docs/task.md` の「優先課題」または「一般課題」セクションに目次のみを登録します（例: `- [ ] **Phase XX-Y: タイトル** (#15)`）。
3. **計画書作成**: 実装着手時に `docs/plans/` 配下に計画書（`YYYY-MM-DD-計画書名.md`）を `[ACTIVE]` タグ付きで作成します。

### 5-2. 課題（バグ修正・要改善項目）の追加
1. **GitHub Issue 起票**: エラーログや再現手順を記載して GitHub 上で Issue を作成し、`ISSUE-XX` の管理番号と Issue 番号（例: `#42`）を確定させます。
2. **タスク登録**: `docs/task.md` の「バグ修正」または「保留案件」セクションに目次のみを登録します（例: `- [ ] **ISSUE-XX: タイトル** (#42)`）。
3. **計画書作成**: 修正着手時に `docs/plans/` 配下に計画書を作成し、バグを再現するためのテストコード（`#[test]`）の追加計画を必ず含めます。

## 6. 本本番デプロイ時の安全チェック（デプロイ前 DoD）

本番機（`rp1`）に不具合のあるコードがデプロイされるのを防ぐため、`scripts/deploy.sh` を実行する前に必ず以下のチェックを行います。

1. **ブランチの確認**: 現在のブランチが `main` であること。
2. **未コミット変更の確認**: ワーキングツリーに未コミットの変更（Modified）が残っていないこと（`git status --porcelain` が空であること）。
3. **リモート同期の確認**: ローカルの `main` がリモート `origin/main`（CI検証済み）と完全に同期していること。
