## 1. 概要

実運用環境（RPi4 / RAM 8GB / ローカル CPU 稼働）において、計算・メモリ負荷を最小限に抑えつつ、エージェントの行動制御精度と知的応答精度を最大化するための、各種マークダウン（MD）ファイルの物理的取り扱い・RAG インジェスト・固定コンテキスト構築の方針を定義する。

## 3. 親子チャンキング (Parent-Child Chunking) 規則

RAG にインジェストするマークダウンファイルのうち、親子関係が `あり` と定義されているものについては、以下の規則で `memory.db` に格納する。

1.  **親チャンク (Parent Chunk)**:
    *   Markdown の見出し（`##`, `###`）から次の見出しまでの区切り、または 1,000〜3,000 文字の論理的な段落ブロック。
    *   SQLite の `embeddings` テーブルに `text` のみ格納し、`vector` (Embedding) は生成しない（Null）。`parent_id = NULL`。
2.  **子チャンク (Child Chunk)**:
    *   親チャンク内の箇条書き（`- `）の各行、または 100〜300 文字の短い文。
    *   ローカル Embedding モデルでベクトルを生成し、親チャンクの主キー（ID）を `parent_id` カラムに設定して SQLite に格納する。
3.  **引き当て (Retrieval)**:
    *   子チャンクがベクトル類似検索（コサイン類似度 >= 0.60）でヒットした際、`parent_id` を介して親チャンクのテキスト全体を引き抜き、システムプロンプトに動的注入する。

## 4. ライフサイクル & クリーンアップ規則

### 4.1. 生ログ (memory/logs/*.md) のプルーニング
*   `memory/logs/` 内の生ログファイルは、ファイル作成日または最終更新日から 14 日以上経過したものを、`ingest_session_summary` のクリーンアップバッチ時に RAG インデックス（SQLite の `embeddings` 内の該当レコード）から削除する。

### 4.2. 監視結果 (patrol/findings.md) の上書き
*   新たな Patrol 監視結果が書き込まれると、インジェスト処理時に前回の `source_id = 'doc:patrol/findings.md'` に紐づく全レコードを SQLite から物理削除し、新規の findings のみをインジェストする。
