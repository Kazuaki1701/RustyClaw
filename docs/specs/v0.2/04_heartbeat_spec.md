# 04. Heartbeat（自発行動）システム仕様

> [!NOTE]
> **ステータス**: `[ACTIVE]` (最新の真実 - コードと同期中)  
> **最終更新日**: 2026-06-08  
> **対象コード**: `rustyclaw-gateway` (HeartbeatService) の最新実装

## 1. 概要とファイルの役割

Heartbeatシステムは、ユーザーからの入力がない状態でも、エージェントが定期的にバックグラウンドでシステム、カレンダー、通知等を自発的に巡回・点検し、必要に応じて自発的に声掛けやメモリ整理を行うための仕組みです。

### 制御ファイルとデータベースの役割

| ファイル / データベース | 役割 | 変更権限者 |
|---|---|---|
| `HEARTBEAT.md` | Heartbeat実行時の基本行動指示書（静的）。 | **ユーザーのみ**（自己改変禁止） |
| `USER.md` 内 `## Interests` | 興味関心・パトロール対象の定義。 | ユーザー or エージェント |
| `memory/heartbeat-state.json` | 各種チェック（Step）の最終実行時刻。 | エージェント（自己更新） |
| SQLite `patrol_state` テーブル | `heartbeat-state.json` のRust側状態管理。 | システム（自動） |
| SQLite `seen_items` テーブル | 重複通知を防ぐためのパトロール既読管理。 | システム（自動） |

## 2. HEARTBEAT.md の 7 Step 構造

Heartbeat実行時にエージェントが順に処理する 7 ステップです。GeminiClawの原版仕様を流用します。

| Step | 実行頻度 | 処理内容 |
|---|---|---|
| **Step 1** | 毎回 | `heartbeat-digest.md`、`summaries/`、`logs/` から前回の巡回以降の全活動を把握し、文脈を認識する。 |
| **Step 2** | 数時間ごと | `MEMORY.md` の整理、`USER.md` 内 `## Interests` セクションの更新・クリーンアップ。 |
| **Step 3** | 毎回 | 外部カレンダー・メール等の着信チェック（ツールが実装されていない場合は無言でスキップ）。 |
| **Step 4** | 1日2〜3回 | 天気予報チェック（4時間インターバル推奨。ツール未実装時はスキップ）。 |
| **Step 5** | 毎回 | ユーザーとの最終対話から **8時間以上経過** しており、かつ現在が昼間（Quiet hours: 23:00〜08:00 は除外）の場合、文脈に沿って自発的な軽い声掛けを行う。 |
| **Step 6** | ローテーション | 未完了タスクの消化、失敗したセッションの確認・対処、バックグラウンドでの自発的なコード改善等のタスク実行。 |
| **Step 7** | 毎回 | パトロール結果を評価し、完了時に **必ず `HEARTBEAT_OK` を含めて応答** する。 |

### 💡 実装完了ステータス
現在、すべてのステップ (Step 1〜7) が完全に実装され稼働しています。
*   **Step 3 (カレンダー・メールチェック)**: Google Workspace 連携ツール (`gws`) を用いたスケジュールと新着メールの取得が実装されており、既読判定には SQLite `seen_items` テーブルを使用しています。
*   **Step 4 (天気予報・雨雲パトロール)**: Open-Meteo API を用いた 60分先降水量予測（雨雲パトロール）が実装されており、雨の接近を検知した場合は `HEARTBEAT_OK` の出力を抑制し、ユーザーへ雨雲アラートを通知します。
*   **Step 6 (自発作業)**: Obsidian の書き込み・読み込みや各種スクリプトの実行など、登録された MCP ツールを介して様々な自発作業が行われます。

## 3. Heartbeat 応答規約 (HEARTBEAT_OK)

エージェントはHeartbeatパトロールを完了した際、結果の重要度（Critical）に基づいて配信か無音かを判断します。

| 重要度レベル | 条件 | 処理 | `HEARTBEAT_OK` の出力 |
|---|---|---|:---:|
| **Critical**<br>(緊急アラート) | 緊急メール、直近のデッドライン、重大障害、および Step 5 による「ユーザーへの声掛け」が発生した場合。 | ユーザーへプッシュ通知（テキストとして配信）を行う。 | **No** |
| **Informational**<br>(定常通知) | 新着メッセージはあるが緊急でない場合、予定の定常確認など。 | ユーザーには通知せず、`logs/YYYY-MM-DD.md` に結果を静かに追記する。 | **Yes** |
| **Nothing**<br>(異常なし) | パトロールした結果、特にアクションが必要なかった場合。 | 何もせず無音で処理を終了する。 | **Yes** |

## 4. 設計・実装上の重要ルール

### ① HEARTBEAT_OK による「無音」制御
システムはエージェントの応答テキストに `HEARTBEAT_OK` が含まれている場合、**ユーザーに対して一切の配信（Telegram等のメッセージ送信）を行わず、プロセスを正常終了** させなければなりません。

### ② 自発活動による「アクティブ時刻」の保護
Heartbeat実行によるAPI呼び出しおよびメッセージ送信は、**`last_user_interaction_at`（最終ユーザー対話時刻）を絶対に更新してはなりません**。
もし更新してしまうと、エージェント自身の発言によって「対話があった」とシステムが誤認し、Step 5 の「8時間以上無通信による声掛け」が永久にトリガーされなくなるためです。

### ③ HEARTBEAT.md の自己改変禁止
エージェントは自発活動やユーザーとの会話を通じて、様々なファイルを自己改変して学習しますが、**`HEARTBEAT.md` のファイル内容だけは絶対に自己改変してはなりません**。

### ④ 既読管理のデータベース化
同一のメールやRSSなどのソースから重複して声掛けが発生するのを防ぐため、既読管理は `seen_items` テーブルで行います。ファイルに状態を追記することによるSSD書き込みのスパイクを避けます。

### ⑤ キューの最大数制限（バックグラウンド枠）
Heartbeatセッションのキューが処理遅延などで積み重なるのを防ぐため、バックグラウンド実行レーンのキューは最大で **1件** に制限します。新しいHeartbeatリクエストが来た際、古い未処理のキューは破棄します。

## 5. Heartbeat ループのコンテキスト制御（オーバーフロー対策）

Heartbeat ループでは、ツール呼び出しを繰り返すたびにメッセージ履歴が肥大化し、モデルのコンテキスト上限（Groq 6,000 トークン等）を超過するリスクがあります。
Phase 28b-4 でこれを防ぐための 2 層のキャップ機構を実装しました。

### ① ツール結果の truncate（`truncate_70_20`）
ツール呼び出し結果を履歴へ追加する際、`truncate_70_20` 関数により **3,000 bytes** にキャップします。
先頭 70% + 末尾 20% を保持し、中間を省略記号（`[... truncated ...]`）で置換します。

### ② 世代ローテーション（廃止）
過去の実装（Phase 28b-4）では `trim_heartbeat_messages` を適用し、直近 1 世代分を除く assistant/tool メッセージを破棄していました。
しかし、これによってカレンダーやメールなどの複数のツールを順次実行した際に過去の実行結果のコンテキストを忘れてしまい、再実行ループ（上限5回でクラッシュ）を引き起こすバグ（ISSUE-26）が判明したため、本機構は廃止されました。
現在は `truncate_70_20` および RAG 注入量の最適化により、履歴全体を安全に維持しつつ複数ツールの連続実行が可能です。

## 6. `heartbeat-digest.md` の生成ロジック（バックグラウンド Quota 対策）

Heartbeatを実行する直前に、システムが過去の会話ログをスマートに要約圧縮し、一時入力として自動生成するインプットファイルです。

### 生成アルゴリズムとトークン節約の仕組み
1. **増分スキャンとディープスキャンのハイブリッド方式**:
   - 平時は**増分スキャン**を行い、前回巡回時刻（SQLite の `heartbeat_last_run_ts`）以降に更新されたアクティブセッション（過去24時間以内）のみ JSONL をロードして解析します。
   - 前回から更新のないセッションについては、既存の `heartbeat-digest.md` ファイルから対応する行（`[--:--] {session_id}: ...`）をパースしてインメモリで再利用し、重複するディスクI/Oとデシリアライズコストを削減します。
   - カウンタ（`heartbeat_run_count`）が 6 の倍数に達するごとに、過去24時間以内にアクティブだった全セッションログを強制的に再スキャンする**ディープスキャン**を起動し、ログデータの整合性を担保します。
2. **自己参照除外フィルタ**:
   - `cron-heartbeat.jsonl` （過去のHeartbeat実行ログ自体）はスキャン対象から完全に除外します。自発アクションによる自己参照無限ループと無駄なトークン消費を防止します。
3. **厳格なサイズ制限**:
   - ダイジェストは **最大3000文字（約600〜800トークン）** 以内に収まるよう、最新のエントリを優先して厳格にクリップ（圧縮）します。
4. **超高密度フォーマット**:
   - 各対話ログを `[--:--] {session}: {prompt} → {response}` という極めて無駄のない1行フォーマットに圧縮整形し、`heartbeat-digest.md` に原子性書き込みします。

## 7. Heartbeat RAG 最適化（Phase 42）

### ① RAG クエリ生成（Phase 42-A）

Heartbeat 実行直前に `build_heartbeat_rag_query()` が RAG 検索用クエリを生成します。

- `heartbeat-digest.md` の**末尾 `HEARTBEAT_RAG_TAIL_LINES`（= 10）行**を取得し、固定プレフィクス `"recent errors tasks memory updates: "` と連結する
- digest が存在しない場合はプレフィクスのみをクエリとして使用する
- 生成されたクエリは `retrieve_rag_context_local` / `retrieve_rag_context` に渡される

### ② ステップ別 RAG 注入（Phase 42-B）

`execute_heartbeat()` は RAG コンテキストを **3 種類のクエリ**で取得し、関連するステップに注入します。

| 定数 | クエリ内容 | 注入先 |
|---|---|---|
| `effective_rag`（42-A 由来） | digest 末尾 + 固定テンプレート | システムコンテキスト全体 |
| `HEARTBEAT_STEP2_RAG_QUERY` | `"user interests hobbies routine habits long-term memory"` | Step 2（ユーザー状況把握） |
| `HEARTBEAT_STEP6_RAG_QUERY` | `"errors bugs pending tasks todo improvements"` | Step 6（タスク・改善確認） |

ローカル embedding（`use_local_embedding: true`）とリモート RAG（`UnifiedRagEngine`）の両方に対して同様に 3 クエリを発行します。

### ③ システムコンテキストの静的化とキャッシュ最適化（Phase 42-C）

プロンプトキャッシュのヒット率を最大化するため、`build_heartbeat_context()` は**静的コンテンツのみ**を返します。

- **`build_heartbeat_context()`**: `SOUL.md` + `HEARTBEAT.md` のみを読み込んで返す（動的コンテンツを含まない）
- **`execute_heartbeat()`**: `build_heartbeat_context()` の結果に `[now: {timestamp}]` と RAG チャンクを追加する

この構造により、静的な `SOUL.md` + `HEARTBEAT.md` 部分がキャッシュに乗り、動的な `[now:]` と RAG 注入のみが毎回新規トークンとして送信されます。

### ④ 時間減衰リランキング（Phase 42-D）

RAG 検索時に `EmbeddingConfig.time_decay_half_life_days` が設定されている場合、コサイン類似度に経過時間ペナルティを乗じてリランキングします。

```
combined_score = cosine_sim × 0.5^(age_days / half_life_days)
```

- **`half_life_days` 未設定**（`None`）: 従来のコサイン類似度のみでランキング（後方互換）
- **`half_life_days` 設定済み**: `search_similar_with_decay()` を使用し、古い記憶より直近の記憶を優先
- threshold はコサイン類似度に対して適用（relevance ゲート）
- `created_at` のパース失敗時は `age_days = 0.0`（fail-open、decay なし）

設定例（`config.json`）:
```json
"embedding": {
    "use_local_embedding": true,
    "time_decay_half_life_days": 30.0
}
```
