# RustyClaw リクエストプロンプト（指示文・スキル定義）最適化検討報告書

> [!NOTE]
> **ステータス**: `[ACTIVE]` (Phase 52-2 実装の設計資料)  
> **最終更新日**: 2026-06-13

## 1. 目的と現状の課題
06:36:18 の対話ダンプ（`デイリーブリーフィングをお願い`）において、ユーザーの短い発話に対して入力トークン数が **10,030 トークン** に達している現象を精査しました。
リクエスト文とレスポンスを対比・精査し、トークン無駄遣いの主因を特定し、最適化の改善案を提示します。

---

## 2. リクエストとレスポンスの対比分析
今回の取引におけるデータ対比は以下の通りです。

| 項目 | データサイズ / 内容 | 役割・課題点 |
| :--- | :--- | :--- |
| **リクエスト: `SOUL.md`** | 約 5.2 KB (5,275 bytes) | **固定ペルソナ**: 秘書人格の設定や厳格なルール。毎回丸ごと送信されており、削減余地あり。 |
| **リクエスト: `USER.md`** | 約 3.5 KB (3,505 bytes) | **ユーザー背景**: 興味関心（Obsidian, HA, Llama等）や通勤ルート。特定の会話（天気やメールなど）ではほぼ不要なデータが混入。 |

| **レスポンス: `tool_calls`** | 4つの並列ツール呼出 | `vitals`, `calendar`, `weather`, `gmail` の実行指示。入力の9割以上を占める他のスキル情報は未使用。 |

## 3. 具体的な最適化改善案

### 3.1. 改善案 A: 【スキル定義】動的スキル選択（Dynamic Skill Selection）の導入
現在、すべての実行可能スキル一覧（Obsidian, HA, Weather, Calendar等）を毎ターン無条件で LLM に送信しています。これをユーザーのク脈に応じて前処理でフィルタリングします。

*   **動作イメージ**:
    1.  ユーザーが「ブリーフィングをお願い」または「予定は？」と入力。
    2.  Gateway 側でキーワード（予定、メール、天気、バイタル）判定、あるいは **MCP `context-mode` の検索機能（`ctx_search` による類似度抽出）** を用いて、関連するスキル（`calendar`, `gmail`, `weather`, `vitals-coach`）のみを抽出。
    3.  抽出されたスキル定義のみを `user` プロンプトに添付する。
*   **期待効果**: スキル記述テキストの **70〜90% を削減**（毎リクエストあたり **約 2,000〜2,500 トークン節約**）。

### 3.2. 改善案 B: 【USER.md】興味関心（Interests）等の動的読み込み
`USER.md` のうち、ユーザーの「プロフィール基本情報」や「会話のルール」は対話に必須ですが、「興味関心（Obsidian, Cloudflare などのソースリスト）」は会話の中でそれらのトピックが出ない限り不要です。

*   **改善アプローチ**:
    *   `USER.md` を「基本・優先ルール」と「長期興味（Interests / Notes）」に分割。
    *   「長期興味」の情報を **MCP `context-mode` 側のドキュメントデータストアにインデックス登録**しておき、会話内で該当ワード（例: 「Obsidian」「ローカルLLM」等）が検知された場合、または **`ctx_search` によるベクトル検索にヒットした場合のみ**、システムプロンプトに動的に注入する。
*   **期待効果**: `USER.md` の送信サイズを **約 50% 削減**（毎リクエストあたり **約 1,500 トークン節約**）。

### 3.3. 改善案 C: 【スキル設計】スクリプトインターフェースの集約
Home Assistant などの同一機能群で複数のスクリプト（`201_ha_logs.sh` や `202_ha_health.sh` など）が露出しているため、スキル定義文が肥大化しています。

*   **改善アプローチ**:
    *   露出するスクリプトを `skills/home-assistant-rest-api/scripts/ha-control.sh` の1本に集約し、引数（`logs`, `health`, `report` 等）で制御する設計に統合する。
    *   LLM に提示する定義行数は 1本分で済むため、記述量が劇的に減少。
*   **期待効果**: スキル定義テキストを **約 800 トークン節約**。

### 3.4. 改善案 D: 【スキル動作】外部スクリプトの MCP ツール（context-mode ネイティブ）化
現在、カレンダー、天気、バイタル、Gmail などの各スキルは `run_workspace_script`（実体は `ctx_execute`）を用いてシェルスクリプトを実行していますが、これはプロセス起動（fork）のオーバーヘッドやスクリプトパス記述の冗長さを伴います。

*   **改善アプローチ**:
    *   これらのスクリプトによるデータ取得ロジックを、**MCP `context-mode` 自体のカスタム MCP ツール（TypeScript/Node.js実装）として直接統合**します（例: `get_calendar_events`, `get_gmail_messages` 等）。
    *   これにより、LLM はシェルスクリプト実行パスではなく、直接的な MCP ツールとしてこれらを呼び出せるようになり、記述の簡素化（トークン削減）とプロセス起動なしでの高速実行が可能になります。

### 3.5. 改善案 E: 【ブリーフィング高度化】エピソード記憶との相関検索および格納
デイリーブリーフィング生成時において、取得した最新のバイタル情報や予定情報に対し、過去のコンテキストを重ね合わせます。

*   **改善アプローチ**:
    1.  バイタルで睡眠スコアの低下を検知した際、自動的に **`ctx_search` を用いて「過去に睡眠不足だった日の K様の体調や行動傾向」を検索**し、アドバイスとしてブリーフィングに動的に付与します。
    2.  生成されたデイリーブリーフィングの最終結果テキストを、完了時に **`ctx_index` ツールを用いてエピソード記憶に即座に登録**します。これにより、「先週の体調はどうだったか？」などの後続の問い合わせに対して、LLM が `ctx_search` で即座に過去データを正確に検索できるようになります。

## 4. 総予測効果

上記改善案の適用により、チャット開始時の初期入力トークンサイズは以下のように劇的に削減されます。

| 項目 | 適用前 | 適用後 (動的最適化) |
| :--- | :---: | :---: |
| **Available Skills** | 3,000 tokens | 600 tokens |
| **USER.md (Interests除外)** | 3,000 tokens | 1,200 tokens |
| **SOUL.md (プロンプト圧縮)** | 4,000 tokens | 2,500 tokens |
| **初期入力合計** | **10,000 tokens** | **4,300 tokens (57% 削減)** |

入力トークン数の削減は、ローカル LLM 動作環境での処理速度（Time to First Token）を大幅に向上させ、メモリ消費量を抑える直結策となります。
