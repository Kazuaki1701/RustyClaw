# Phase 43-D 設計書: RAG-based Guardrails — 実運用ルールの動的適用

**日付**: 2026-06-07  
**案件番号**: Phase 43-D  
**ステータス**: 設計承認待ち  

---

## 1. 概要

実運用環境（RPi4 でのボット常駐監視・チャット対応）において、エージェントボット自身の行動制御やスキル呼び出しの規約（`skills/**/*.md` や `USER.md` 内の運用ガイドライン）を RAG のインジェスト対象として最適化し、対話やタスク実行時に必要な運用ルールだけを自動的に検索してシステムプロンプトに動的注入する。  
開発者向けのルール（`ai-rules.md` や `docs/specs/*.md` などの開発・マージ仕様書）は**インジェスト対象から完全に除外し**、実運用に必要なルールのみを最小限のトークン消費で厳密に遵守させる。

---

## 2. 背景

現在、エージェントの運用ルールは以下の課題を抱えている：
*   **不要データの混入防止**: 実運用環境において、エージェントはコードを書いたりコミットしたりしないため、開発ルール（`ai-rules.md` や `docs/specs/*.md`）を参照する必要は一切ない。これらをインジェストすると、検索ノイズとトークン消費の増加を招く。
*   **運用スキルの増加**: 一方で、天気情報取得、健康監視、ディスク空き容量チェックなどの「実運用で呼び出すべきスキル（`skills/**/*.md`）」や「ユーザーごとの連絡・対応ルール（`USER.md`）」は、エージェントが正しく解釈して行動するために必須である。
*   **プロンプトの動的最適化**: これら増加する実運用手順やスキル仕様をすべて固定プロンプトで毎回の対話に注入すると、コンテキストが肥大化し RPi4 環境での動作遅延を招く。

したがって、開発ドキュメントを完全に排し、**実運用に関するスキル・動作ガイドラインに特化した動的適用（RAG-based Guardrails）** を実装する。

---

## 3. アーキテクチャ

既存のドキュメント・インジェスト機能 (`ingest_static_documents`) を見直し、実運用スキルおよび運用ガイドファイルのみを RAG データベースに取り込み、既存の RAG 検索・プロンプト注入機構と統合する。

```
[運用関連ファイル群]
  ├── USER.md               ───┐
  ├── skills/**/*.md (運用)  ───┼─► ingest_static_documents() ─► [memory.db]
  └── AGENTS.md (エージェント) ─┘
                                      │
                                      ▼
[会話・Task クエリ] ──────────► retrieve_rag_context() ────► [動的運用ルール & 記憶]
                                                                      │
                                                                      ▼
                                                            システムプロンプトへ注入
```

---

## 4. 詳細設計

### 4.1. インジェスト対象の限定と除外ルール

`crates/rustyclaw-agent/src/lib.rs` の `ingest_static_documents` 関数において、以下のファイルを対象とする：

*   **インジェスト対象 (実運用に必要なもののみ)**:
    *   `/workspace/AGENTS.md` (ペルソナ・エージェント設定定義)
    *   `/workspace/USER.md` (ユーザー側の運用ガイドラインやプロファイル)
    *   `/workspace/skills/**/*.md` (ボットが実行可能な外部ツールやスクリプトの使用方法・規約)
*   **インジェスト対象外 (実運用で参照不要なもの - 完全除外)**:
    *   `/workspace/ai-rules.md` (開発者用規約)
    *   `/workspace/docs/` ディレクトリ配下の全ファイル（`docs/specs/*.md` などの開発仕様書）

*※注: 現状の `ingest_static_documents` はすでに `AGENTS.md`, `USER.md`, `skills/**/*.md` のスキャンに限定されているため、コード側で開発用ファイルをインジェストしないよう除外状態を維持・確定させ、RAG 検索側のクエリ・類似度閾値の最適化に注力する。*

### 4.2. 検索動作

RAG 検索クエリ（対話履歴 + 新規入力メッセージから構築されたクエリ）が実行された際：
1. 対話の文脈に関連する運用ガイド（例: 「天気を教えて」という指示に対して、`skills/weather.md` 内のパラメータ指定や動作規約）が上位で類似ヒットする。
2. コサイン類似度が `similarity_threshold` (デフォルト: 0.60) 以上のチャンクが抽出される。
3. 自動的に `system_context` に追加され、LLM は現在のタスクに関連する運用ガイドのみを読み込んだ状態で、スキルツールを呼び出して処理を開始する。

---

## 5. テスト・検証方針

1.  **インジェスト機能の確認**:
    *   開発仕様書 (`docs/specs/*.md`) や `ai-rules.md` がインジェスト対象に入っておらず、`memory.db` にそれらの embedding が存在しないことを確認する。
2.  **スキル引き当てテスト**:
    *   "weather forecast" などのキーワードで RAG 検索をエミュレートし、`skills/weather.md` 由来のチャンクが正しく上位取得できるかアサーションテストを行う。

---

## 6. スコープ外

*   開発用ドキュメントの検索サポート（実運用に不要なため、今後も実装しない）。
*   親子関係付きチャンキング（Parent-Child Chunking）の導入（Phase 43-E で別途対応）。
