# `PreCompact / SessionStart` フックによる状態スナップショット記憶継続の検証 ＆ 計画書

> **ステータス**: `[PLANNING]`  
> **作成日**: 2026-06-13  
> **関連**: [`docs/plans/2026-06-13-context-optimization-proposal.md`](2026-06-13-context-optimization-proposal.md)  
> **対象案件**: `Phase 53-2` (将来実装候補)

---

## 1. 検証の目的と背景
現在、RustyClaw v0.4 は会話履歴が一定のトークン上限（または TPM Cap）に近づいた際、`compact_if_needed` によって **70/20/10 戦略（中間メッセージの省略）** を実行し、コンテキストの破綻を防いでいます。

しかし、この単純な中間メッセージの削除（切り詰め）は以下の問題を引き起こします。
1. **中間ステップでの試行錯誤やデバッグ結果の消失**: 「〇〇を試したが動かなかった」という知見が消え、エージェントが同じエラー修正アプローチを繰り返す。
2. **合意した設計方針・仕様の忘却**: 最初の数ターンでユーザーと決定した制約条件が中間からパージされ、矛盾した実装を提案する。

本計画書は、`context-mode` の `PreCompact` および `SessionStart` フックを模した**「半自律的状態スナップショット（Session Guide）」**を RustyClaw に統合し、中間コンテキストを 80% 以上削減しつつ、記憶の継続性を最大化するための具体的な活用場面と仕様を定義・検証します。

## 2. 具体的な活用場面（ユースケース）

### 2.1 ユースケースA：長尺のリファクタリング ＆ ビルドエラー解消
*   **状況**: エージェントが 3 つ以上のクレートにまたがる大きな機能修正（例: `rustyclaw-gateway` と `rustyclaw-agent` 間の API 変更）を行い、コンパイルエラーや Clippy 警告の修正のために 15 ターン以上対話が続く。
*   **課題 (記憶喪失)**: 
    - 12 ターン目にコンパクションが走り、4〜10 ターン目の「どのファイルをどう直したか」「どのエラーが解決済みで、何が未解決か」のログがシステムメッセージ `[7 messages omitted...]` に置き換わる。
    - エージェントは解決済みのファイルに再び余計な修正を加えたり、エラーの原因を誤認してループに陥る。
*   **解決フロー**:
    1. **`PreCompact` 検知**: トークン超過による履歴圧縮の直前、エージェント（または裏で動く `context-mode`）にこれまでの履歴を要約させ、「現在変更中のファイル」「解決したビルドエラー」「現在の残タスク」を XML 形式で抽出する。
    2. **`SessionStart` 注入**: 圧縮後の履歴の先頭に、このスナップショットを「Session Guide」として挿入する。

---

### 2.2 ユースケースB：定常パトロール中のマルチステップ障害原因調査
*   **状況**: 毎時実行される cron スクリプト（例: `203_ha_summary.sh`）で外部 API 接続エラーが発生。エージェントが自律的にコンソールログ確認、コンフィグ点検、テスト実行などを順に行う。
*   **課題 (調査ループ)**:
    - 調査ステップが 10 ターンを超えるとコンパクションが発生。
    - 「3 ターン目に DB の接続確認をして正常だった」「5 ターン目に vault のシークレット移行を確認して問題なかった」といった「正常だった箇所（容疑から外れたコンポーネント）」の記憶が消える。
    - 結果として、エージェントはすでに確認済みの DB 接続や vault 設定を何度も点検し直し、トークンと実行時間を浪費する。
*   **解決フロー**:
    - `PreCompact` フックで「調査済みコンポーネント一覧と結果（ステータス：PASS/FAIL）」および「現在の調査容疑リスト」をスナップショットに退避。次のターンでは「DB接続確認はパス済み」という前提知識を持ってログ探索を継続する。

---

### 2.3 ユースケースC：`grill-me` による対話型要件定義・設計ブレスト
*   **状況**: 新しいチャンネル（例: LINE チャンネル）の設計にあたり、ユーザーとエージェントが「通知の頻度」「例外発生時のロールバック仕様」「認証の有無」などについてブレインストーミングを行う。
*   **課題 (仕様の不整合)**:
    - 2ターン目に「今回は通知の頻度は10分に1回を上限とする」と合意したにもかかわらず、12 ターン目の実装フェーズ（コンパクション後）でエージェントが「即時通知を行うコード」を書いてしまう。
*   **解決フロー**:
    - `PreCompact` 時点までに確定した「仕様合意事項（Constraints）」と、逆に却下された「除外要件（Non-Goals）」をスナップショットに固定。LLM は常にこの固定された制約を参照してコードを生成する。

---

### 2.4 ユースケースD：日またぎ時のタスク継続
*   **状況**: 日付が変わったタイミングでセッションが新しく切り替わる、または新しい日付のセッション（例: `cron:2026-06-14`）として監視・パトロールタスクが立ち上がる。
*   **課題 (コールドスタート問題)**:
    - 日付変更に伴い、会話履歴は初期化される（または大幅に削減された状態からスタートする）ため、前日の詳細な試行錯誤（ビルド結果やデバッグ中のエラーログ等）は持ち越されない。
    - エージェントは「昨日どこまで作業が進み、今日何から再開すべきか」が分からず、無駄な再調査から始めるか、あるいはユーザーに作業再開箇所を質問するコールドスタート状態になる。
*   **解決フロー**:
    - 前日の最終実行（またはセッション終了）時に生成された最後の `session_guide` (状態スナップショット) を DB に永続保存しておく。
    - 日またぎ後最初のコンテキスト構築時 (`SessionStart` / 新日初回起動フック) において、この永続化された前日最終のスナップショットをシステムコンテキストに自動マウントする。これにより、履歴がクリアされた状態でも「前日の終了ステータス」と「今日のToDo」を即座に把握し、ウォームスタートを可能にする。

## 3. LLM Request 時のコンテキスト構築における統合・活用シーン
LLM への PromptRequest を組み立てる「コンテキスト構築フェーズ」において、状態スナップショット（XML）は以下のように動的かつセキュアに注入・活用されます。

### 3.1 活用シーン①：Chat対話での System Preamble への動的注入
- **コードパス**: `rustyclaw-agent` 内の [build_system_context](file:///home/kazuaki/Projects/RustyClaw/crates/rustyclaw-agent/src/lib.rs#L506)
- **仕組み**: 
  - ユーザーメッセージを受け取って LLM Request を組み立てる際、セッション履歴に紐づく最新の `session_guide` (XML) が存在すれば、それを `USER.md` や `SOUL.md` と並列のシステムプロンプトヘッダーとして動的にインジェクションします。
  - これにより、メッセージ履歴（`rig_history`）を直近 2〜3 ターンに極限まで削減しても、合意事項や残タスクを完璧に維持したリクエストを構成できます。

### 3.2 活用シーン②：Heartbeat パトロール中の「調査コンテキスト」の伝播
- **コードパス**: `rustyclaw-agent` 内の [build_heartbeat_context](file:///home/kazuaki/Projects/RustyClaw/crates/rustyclaw-agent/src/lib.rs#L1007) および [execute_heartbeat](file:///home/kazuaki/Projects/RustyClaw/crates/rustyclaw-agent/src/lib.rs#L1033) のループ処理
- **仕組み**:
  - 監視タスクでは、ログやデバイス状態を取得するツール出力が大量に発生します。これらが 3,000 バイトを超えコンパクションされた際、ラッパーレベルで「どのチェックが正常終了したか」という状態スナップショットを生成します。
  - 次のループ実行時の `preamble` にこのスナップショットを注入することで、LLM は「前回のループでDB接続とConfigは確認済みである」という知識を最初から持った状態で、無駄な検証を繰り返すことなく動作できます。

### 3.3 活用シーン③：RAG クエリ抽出の「文脈キーワード」としての利用
- **コードパス**: `rustyclaw-agent` 内の RAG 検索クエリ構築 ([build_heartbeat_rag_query](file:///home/kazuaki/Projects/RustyClaw/crates/rustyclaw-agent/src/lib.rs#L144) 等)
- **仕組み**:
  - 単純に直近のメッセージだけで RAG を引くと、前後の文脈が抜けてノイズの多い類似箇所がヒットし、コンテキスト枠を無駄に埋めてしまいます。
  - `session_guide` にある `current_goal` や `resolved_errors`（エラーIDなど）をクエリキーワードとして自動合成して RAG を引くことで、現在のタスクに 100% 直結した過去記憶（例：「3日前に同様の接続エラーを解消した時のログ」など）だけをピンポイントで引き当て、コンテキスト構築の効率を最大化します。

### 3.4 活用シーン④：日またぎ（日付変更）時のウォームスタート
- **コードパス**: `rustyclaw-gateway` および `rustyclaw-agent` 内のセッション初期化処理（`SessionStart` フック相当）
- **仕組み**:
  - 日付変更後、その日最初のリクエスト（会話履歴メッセージがほぼ空の状態）で LLM Request が作られます。
  - この時、セッション初期化ロジックが `context-mode` の FTS5 データベースを検索（`context-mode search` 経由、または直接 MCP `ctx_search` を介して最新の `session_guide` をクエリ）し、永続保存されている前日最終の `session_guide` を引き当てて読み込み、システムコンテキスト (`preamble`) にインジェクションします。
- **効果**:
  - 前日の対話履歴（数万トークン）を全てロードすることによるトークン浪費と読み込み遅延をゼロにしつつ、「昨日はどこのデバッグまで終わり、今日次はどのテストから走らせるべきか」という開発ステートを引き継いだ状態で、新日の1ターン目から正確なタスク実行が再開できます。

## 4. スナップショット (Session Guide) のプロトタイプ
コンパクション時に生成され、履歴の先頭（システムコンテキストの直後）にインジェクションされる XML 構造体のプロトタイプです。

```xml
<session_guide>
  <task_context>
    <current_goal>rustyclaw-gateway への LINE チャンネル統合</current_goal>
    <phase>IMPLEMENTING_CONNECTOR</phase>
    <status>ACTIVE</status>
  </task_context>
  
  <checkpoint>
    <resolved_errors>
      <error id="E0433">crates/rustyclaw-gateway/src/lib.rs での LineConnector の未定義エラー（インポート追加で解決）</error>
    </resolved_errors>
    <verified_components>
      <component path="crates/rustyclaw-config">スキーマ定義は正常にビルド・テスト通過</component>
    </verified_components>
  </checkpoint>

  <design_constraints>
    <constraint>通知頻度は最大 10分に 1回（Spam 防止）</constraint>
    <constraint>認証トークンは必ず `rustyclaw vault` から動的取得する</constraint>
  </design_constraints>

  <next_steps>
    <step order="1">crates/rustyclaw-gateway/src/lib.rs の MessageBus への登録処理の実装</step>
    <step order="2">統合テスト `test_line_connector_flow` の実行とパス</step>
  </next_steps>
</session_guide>
```

## 5. 期待される効果 (定量的シミュレーション)

対話が 20 ターン（通常、コンパクションが 1〜2 回発生する長さ）継続した場合のコンテキスト削減と品質への影響予測：

| 指標 | 従来方式 (単純な 70/20 履歴切り詰め) | スナップショット継続方式 (PreCompact 適用) | 改善率 / メリット |
|:---|:---|:---|:---|
| **平均入力トークン数** (15ターン目) | 約 12,000 tokens | 約 3,500 tokens | **70.8% 削減** |
| **調査ループ発生率** (ユースケースB) | 35% (記憶喪失による重複点検) | 5% 未満 | 調査ターンの無駄な往復を排除 |
| **仕様不整合の発生件数** (ユースケースC) | 平均 1.5 件 / セッション | 0 件 (制約が常に XML に明記) | ハルシネーションの劇的低減 |
| **1ターンあたりの推論遅延** | 平均 6.5 秒 (RPi 4 環境) | 平均 2.1 秒 | **約 67% の高速化** |

## 6. 実装ロードマップ（Phase 53-2）

1. **Step 1 (Snapshot Extractor & context-mode index)**:
   `rustyclaw-storage` に `PreCompact` ハンドラを実装。`compact_if_needed` がトリガーされた際、履歴全体の要約から生成された XML スナップショットを特定のファイル（例: `memory/sessions/YYYY-MM-DD.xml`）に書き出し、直後に `context-mode index <path>` コマンドをサブプロセスまたは MCP 経由で呼び出して FTS5 知識ベースに高速登録（インデックス化）します。
2. **Step 2 (Injection logic & context-mode search)**:
   日またぎ後の初回起動やコンパクション後の最初のリクエスト時、`rustyclaw-agent` 側で `context-mode search`（または直接 `context-mode` 経由の FTS クエリ）を実行し、昨日の最終状態スナップショットを引き当てて取得します。取得した XML データを、[build_system_context](file:///home/kazuaki/Projects/RustyClaw/crates/rustyclaw-agent/src/lib.rs#L506) 内でシステムプロンプトの冒頭に自動注入します。
3. **Step 3 (Validation Test)**:
   意図的にトークンリミットを絞ったテスト用エージェントを稼働させ、`context-mode` 連携による「FTS5 へのインデックス登録 ➔ 日またぎ後の FTS 検索によるスナップショット復元 ➔ コンテキスト自動インジェクト」が一連の動作として警告なしで正常に機能することを検証します。
