## 1. 概要

実運用環境（RPi4）において、検索精度を向上させつつハルシネーション（文脈の欠落）を防ぐために **階層型 RAG (Parent-Child Chunking)** を導入する。  
具体的には、ベクトルデータベース検索には情報密度の高い短い「子チャンク (Child Chunk)」を使用し、ヒット時に SQLite からその周辺の広い文脈を含む「親チャンク (Parent Chunk)」を引き出して LLM のシステムプロンプトに動的注入する。

---

## 2. 背景とメリット

現在の RustyClaw の RAG システムは、ドキュメントを 800 文字単位などで機械的にフラット分割しベクトル化している（フラットチャンキング）。これには以下の課題がある：

*   **文脈の欠落**: 800 文字の境界で文章や箇条書きのリストが途切れるため、エージェントが前後のつながり（例えば、前段にある変数定義や大前提ルール）を見落とす。
*   **コサイン類似度の低下**: チャンクサイズが大きすぎると、特定のコマンドオプションやキーワードの「ベクトル表現」が薄まり、RAG 検索時にヒットしづらくなる。
*   **RPi4 でのトレードオフ**: ベクトル検索用のチャンクを短くすると RPi4 上での検索速度は向上するが、前後の文脈が失われ回答精度が下がる。

階層型 RAG (Parent-Child Chunking) は、**「検索は詳細な子で行い、LLM への提示は文脈を持った親で行う」** ことで、この課題を解決する。

---

## 3. アーキテクチャ

SQLite データベース (`memory.db`) とインジェスト・検索処理を拡張する。

```
1. インジェスト時 (Ingestion)
   [Markdown ドキュメント (例: skills/weather.md)]
          ├── 親チャンク: 見出し単位 (## 雨雲レーダー) ──► SQLite にテキストのみ登録
          └── 子チャンク: 各バレット (パラメータ等) ────► embedding 生成 ＋ parent_id 紐付け ─► SQLite

2. 検索時 (Retrieval)
   [検索クエリ] ──► ベクトル検索 ──► 子チャンクヒット (ID: child_123, parent_id: parent_456)
                                               │
                                               ▼
                                  SQLite より parent_456 のリッチテキストを取得
                                               │
                                               ▼
                                      システムプロンプトに注入
```

## 4. 詳細設計

### 4.1. データベーススキーマ拡張 (`crates/rustyclaw-storage`)

`embeddings` テーブルに、親チャンク ID を指す `parent_id` カラムを追加する。

```sql
-- マイグレーションスクリプト
ALTER TABLE embeddings ADD COLUMN parent_id TEXT DEFAULT NULL;
```

*   `parent_id IS NULL` の場合: そのチャンク自体が独立したドキュメント（親チャンク、またはフラットチャンク）。
*   `parent_id IS NOT NULL` の場合: 他の親チャンクに関連付けられた子チャンク（ベクトル検索の対象）。

`DbManager` の upsert やロード関数において、`parent_id` をオプション引数として扱えるように修正する。

### 4.2. 階層型チャンキングの実装 (`crates/rustyclaw-agent`)

`ingest_static_documents` および `ingest_memory_md` で使用するチャンキングロジックを変更する。

*   **親チャンクの抽出**:
    *   ドキュメントの「見出し (##, ###)」または大きな段落（1,000〜3,000文字）を親チャンクとして抽出。
    *   親チャンクはベクトル化せず、SQLite の `embeddings` テーブルに `parent_id = NULL` としてテキスト情報のみを保存する。
*   **子チャンクの抽出とベクトル化**:
    *   親チャンクの内部をさらに各バレット（`- ` 行）や短い文（100〜300文字）に分割し、子チャンクを作成する。
    *   子チャンクはローカル embedding モデルでベクトル化し、親の ID を `parent_id` に指定して SQLite に格納する。

### 4.3. 検索・引き当てロジック (`retrieve_rag_context`) の修正

検索時に、子チャンクがヒットした場合は親チャンクのテキストをプロンプトに流し込む。

1.  クエリをベクトル化し、データベースでコサイン類似度検索を実行（子チャンクが複数ヒット）。
2.  ヒットした各子レコードの `parent_id` を確認。
3.  `parent_id` が `Some(p_id)` の場合、SQLite から `p_id` のテキスト（親テキスト）を取得する。`None` の場合はその子テキスト自体をそのまま使用する。
4.  **重複排除 (Deduplication)**: 複数の子チャンクが同じ親を指していた場合、親テキストが重複してプロンプトに注入されないように `HashSet` 等でユニーク化する。
5.  フォーマットして `system_context` に結合する。

## 5. テスト・検証方針

1.  **マイグレーションテスト**:
    *   既存の `memory.db` が存在する環境で、エラーなく `parent_id` カラムが追加され、既存のフラットチャンク（`parent_id = NULL` となる）が正常に読み込めること。
2.  **チャンキングの整合性テスト**:
    *   ダミーの Markdown ファイルを読み込み、親子構造が正しく DB に書き込まれることの単体テスト。
3.  **検索テスト**:
    *   子チャンクを類似検索した際、重複が排除された親チャンクのテキスト全体が `retrieve_rag_context` の返却結果に含まれることをアサーション検証。
