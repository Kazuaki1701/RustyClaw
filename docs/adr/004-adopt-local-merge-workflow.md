# ADR 004: ローカルブランチ直行マージ・即デプロイ開発フローの採用

- **ステータス**: `[ACCEPTED]`
- **決定日**: 2026-06-13
- **関連タスク**: Phase 52-2

## 1. コンテキスト（直面した課題）
AI エージェントと K様（ユーザー）による対話型ペアプログラミングにおいて、GitHub 上でのプルリクエスト（PR）起票や、外部 CI サービス（GitHub Actions）のビルド待ち時間が、開発サイクル（実装 ➡ テスト ➡ マージ ➡ デプロイ ➡ ログ点検）全体の速度低下を招いていました。
ローカル開発機での完全自律検証（Clippy と Test）をベースにし、待ち時間なく迅速に実機（RPi4）での動作確認まで到達できる、アジャイルな開発プロセスの確立が求められていました。

## 2. 検討した選択肢（Options）
- **案A: 現行の GitHub PR / CI 依存フローの維持**
  - メリット: リモートでのクリーンな環境検証が徹底され、GitHub UI 上で変更履歴を容易に追跡可能。
  - デメリット: 1コミットごとに数分以上の待ち時間が発生し、対話型開発の機動性が大きく損なわれる。
- **案B: ローカルブランチ直行マージ・即時自動デプロイフロー (本提案・採用案)**
  - メリット: コミットから実機での稼働確認までのフィードバックループが 1〜2 分に短縮。Git hooks による Clippy/テストの自動実行により、ローカルだけでコード品質を 100% 担保可能。
  - デメリット: GitHub PR の会話記録が残らない。

## 3. 決定と選定理由（Decision & Justification）
- **決定**: 案Bを採用し、`AGENT.md` に適用。旧フローは `AGENT_GITHUB.md` としてバックアップ。
- **理由**: ローカルでの厳格なClippy制約と単体テスト（pre-commit / pre-push）でマージ前の安全性は十分に確保できるため、無駄な外部通信・CI待ちを排除し、AI 開発の自律的な推進速度を最大化する方がプロジェクトにとって圧倒的に有利であると判断したため。

## 4. トレードオフ・今後の影響（Consequences）
- 今後は GitHub での PR 作成ステップはバイパスされ、ローカルの `main` で検証が完了した段階でリモートに直接プッシュ（`git push`）します。
- これによりマージ前の「なぜこの変更を行ったのか」という意思決定の背景は、PRの代わりに `docs/plans/` の個別計画書（または本 ADR のようなドキュメント）に責任を持って確実に記述します。
