# ADR 002: Google Workspace (GWS) 連携ツール実装における Shell Script (.sh) 方式の維持

- **ステータス**: `[ACCEPTED]`
- **決定日**: 2026-06-11
- **関連タスク**: BUG-02 / daily-briefing / vitals-night

## 1. コンテキスト（直面した課題）

Google カレンダーや Gmail など、外部 CLI（`gws`）と連携するツールの実装方式について、初期の Rust ネイティブ実装（Rust Native Tool）からシェルスクリプト実行方式（`run_workspace_script` 経由）への移行（Phase 36）を経て、今後の拡張性における優位性を再評価し、どちらの方式を維持すべきかを決定する。

## 2. 検討した選択肢（Options）

### 案A: Shell Script (.sh) 経由での実行 (現行維持)
`run_workspace_script` という汎用の Rust Native Tool を介して、ホスト側の Bash スクリプト（`calendar-ops.sh` 等）を実行する方式。

- **メリット**:
  - **極めて高い開発・修正スピード**: ラズパイなどの本番環境で、Rust コードのコンパイルを必要とせずスクリプトファイルを変更するだけで即座にデバッグ・反映が可能。
  - **外部ツール連携が簡潔**: `gws` の JSON 出力を `jq` でパースして整形するなどのパイプ処理をシェルスクリプトで非常に短く記述できる。
  - **一元化されたセキュリティ**: `run_workspace_script` レベルで Bubblewrap (`bwrap`) サンドボックスを適用しているため、スクリプトごとに個別で保護ロジックを実装する必要がない。
- **デメリット**:
  - **スキーマの隠蔽**: LLM に対する入力パラメータの型（JSON Schema）が `args: Vec<String>` という汎用型になり、LLM が引数構成を間違える可能性が高くなる。
  - **プロセスのオーバーヘッド**: 呼び出しのたびに Bash プロセスをフォークするためミリ秒単位のオーバーヘッドが発生する（ただし、LLM の推論時間と比較すれば無視できるレベル）。

### 案B: Rust Native Tool としての再実装 (直接公開)
`rustyclaw-tools` クレート内で `rig_core::tool::Tool` を実装し、LLM へ直接ツール（`gws_calendar_list_events` 等）として公開する方式。

- **メリット**:
  - **型安全性 (Schema) の担保**: Rust の構造体定義から JSON Schema が自動生成され LLM に渡るため、LLM が引数を誤るリスクが低い。
  - **エラーハンドリングの精密さ**: API レスポンスのエラーなどを Rust 側で細かく制御できる。
- **デメリット**:
  - **修正時のビルドコスト**: 僅かな変更であっても Rust コードの再ビルド・再デプロイが必要になり、開発効率が落ちる。
  - **コードの冗長化**: Rust の `std::process::Command` を用いて外部コマンドやパイプライン処理をエミュレートするため、コードが複雑化・冗長化する。

## 3. 決定と選定理由（Decision & Justification）

- **決定**: **案A: Shell Script (.sh) 方式を維持する。**
- **理由**:
  - `gws` CLI などの外部コマンドラッパーとしての役割が大部分を占めるため、シェルスクリプトによる高い記述性と**「コンパイル不要で即時修正できる俊敏性」**が現時点での最大の強みであると判断。
  - サンドボックス制御を Rust 側の共通基盤に寄せられるセキュリティ上のメリットも大きい。

## 4. トレードオフ・今後の影響（Consequences）

- **LLM の引数誤りへの対策（重要）**:
  LLM が引数（`args`）の構成や順序を間違えやすくなる。このトレードオフを緩和するため、以下の設計を徹底する：
  1. SKILL（`.md` 指示書）におけるパラメータ定義の明記。
  2. スクリプト側での「引数不足時のデフォルトフォールバック」（例: `CMD="${1:-list_family}"`）などのガードロジックの強化。
- **見直しのトリガー**:
  将来的に「外部 CLI（`gws`）のラッパー」から脱却し、Rust 内部のデータベース（SQLite 等）へ複雑な構造化データを書き込む処理や、ミリ秒以下のパフォーマンスが要求される処理が発生した場合は、その一部機能のみを Rust Native Tool として再実装することを検討する。
