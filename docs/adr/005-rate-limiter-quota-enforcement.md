# ADR 005: Rate Limiter クォータ制御方式の設計選択

- **ステータス**: `[ACCEPTED]`
- **決定日**: 2026-06-13
- **関連タスク**: Phase 52（Context 最適化点検中に発覚した実装ギャップの修正）

## 1. コンテキスト（直面した課題）

config.json の `model_list` に `rpm / tpm / rpd / tpd / context_window` が設定されているにもかかわらず、コード点検の結果、以下の未適用・不完全な箇所が判明した。

1. `rpd` / `tpd`（日次上限）— `RateLimitWindow` に日単位フィールドが存在せず、完全に未追跡
2. `tpm`（分単位トークン上限）— `record_tokens()` での warn ログのみ。リクエスト前のチェックなし
3. `execute()` / `execute_heartbeat()` — rpm チェック（`check_rpm()`）が呼ばれていない
4. `tpd` — `record_tokens()` でも未記録

## 2. 検討した選択肢（Options）

### 日次ウィンドウの実装方式

- **案A: プロセス起動相対のインメモリウィンドウ（採用）**
  - メリット: 実装が単純。既存の分ウィンドウ（`minute_start: Instant`）と同一パターン
  - デメリット: プロセス再起動でカウンタがリセットされる。wall-clock 0時に連動しない

- **案B: wall-clock 0時リセット（不採用）**
  - メリット: Groq 等のプロバイダーの実際のリセットタイミング（日本時間 0時）に近い
  - デメリット: タイムゾーン管理が複雑。`chrono` を使った日付比較が必要になり実装コストが増加

- **案C: SQLite 永続化（不採用）**
  - メリット: 再起動をまたいで正確にカウントできる
  - デメリット: `RateLimiter` が DB 依存になり、テストが複雑化。過剰な設計

### rpd 超過時の制御方式

- **案A: warn ログのみ（採用）**
  - メリット: 日次上限超過後に数時間スリープさせることは非現実的。上限超過を記録して次のリクエストは通す
  - デメリット: 実際に超過してもリクエストを止められない

- **案B: 大きなスリープ（不採用）**
  - デメリット: 残り時間（最大 86400 秒）のスリープは現実的でない。`tokio::time::sleep` がブロックしてシステム全体が停止する

- **案C: エラーを返して上位でフォールバック（将来候補）**
  - メリット: 他モデルへのフォールバックが可能
  - デメリット: フォールバック機構の実装が必要で、現時点では過剰

### tpm 超過時の制御方式

- **案A: プリチェック式スリープ（採用）**
  - 実装: `check_rpm()` 内で、リクエスト送信前に分ウィンドウの累積トークンを確認し、超過なら残り秒スリープ
  - メリット: rpm と同一のスリープパターンで一貫性がある
  - デメリット: 前のリクエストのトークン数がないと超過を事前に予測できない（前回 `record_tokens()` した値で判断）

- **案B: ポストチェック warn のみ（旧実装・不採用）**
  - デメリット: 超過しても次のリクエストを止められない

## 3. 決定と選定理由（Decision & Justification）

- **日次ウィンドウ**: 案A（インメモリ・起動相対）を採用。実装コストと信頼性のバランスが最適
- **rpd 超過**: warn のみ。日次上限はプロバイダー側でも 429 が返るため、ソフトな記録で十分
- **tpm 超過**: プリチェック式スリープに昇格（rpm と同一パターンに統一）
- **execute() / execute_heartbeat()**: `check_rpm()` + `record_tokens()` を追加し、全実行パスで rpm/tpm を適用

## 4. トレードオフ・今後の影響（Consequences）

- **プロセス再起動でリセット**: 運用上は問題になりにくいが、連続デプロイ時に rpd/tpd カウントがリセットされる点は注意
- **tpd は warn のみ**: tpd 超過後も逐次リクエストは通る。将来的にフォールバック機構（別モデルへの切替）を実装する場合は制御を強化する
- **ダッシュボード表示**: 日次ウィンドウのスナップショットを `/api/rate-limits` で公開することが次の課題。その際、`RateLimiter` のクレート間公開方式（スナップショット型 vs `Arc<RateLimiter>` 共有）を別 ADR で決定する
