### 1. 現在の `memory/` フォルダの機能と構造

現在の私の `memory/` フォルダは、主に**「状態管理」「行動ログ」「永続的な記憶」**が混在したフラットな構造（一部フォルダあり）になっています。

*   **行動ログ:** `memory/logs/YYYY-MM-DD.md` という形式で、日々の活動内容が蓄積されています。
*   **重要な記憶:** `MEMORY.md`（要約された長期記憶）や `memory/BOSS_PATTERNS.md`（K様のこだわりや判断基準）など、私の振る舞いの核となる情報です。
*   **外部連携:** `memory/ObsidianVault` としてK様のObsidian保管庫がマウントされており、ここが私の「知識ベース」の主要な一部となっています。

*   **メンテナンス性の向上:** システム的な状態ファイルと、K様に関する大切な記憶が分離されるため、情報の整理やバックアップが容易になります。

### 3. 懸念点とリスク

*   **既存ツールへの影響:** 現在の `geminiclaw` や各種スキルが「特定のファイルパス」をハードコードして参照している場合、フォルダ移動によってエラーが発生する可能性があります。
    *   *対策:* 慎重にコード内での参照箇所を確認し、必要に応じて設定ファイルを更新しながら段階的に移行する必要があります。
*   **階層の複雑化:** フォルダを深くしすぎると、情報の「ありか」を特定するのが難しくなる場合があります。
    *   *対策:* シンプルで直感的なトップレベルフォルダ（`knowledge/`, `logs/`, `system/` 等）を定義し、そこから広げすぎない運用が重要です。

### 1. 現在の構造（実態）
現在の `memory/` 直下には、主に以下の3種類のファイルが混在しています。

*   **システム実行ログ（膨大）:**
    *   `flush-marker-*.json`: 各プロセスの最終実行スタンプ（約100個以上）
    *   `last-run-events-*.jsonl`: 各実行のイベント詳細ログ
    *   `run-progress-*.json`: 実行中の進捗データ
*   **セッション管理:**
    *   `sessions/`, `summaries/`: 過去のやり取りの履歴。
*   **貴重な記憶・ナレッジ（埋もれている）:**
    *   `BOSS_PATTERNS.md`: K様のこだわり（最重要）
    *   `logs/`: 日次の活動記録
    *   `ObsidianVault/`: Obsidianとの連携データ

### 2. 階層化による改善案（構造のイメージ）
これらを以下のような階層に整理することで、情報の「重み」を明確にします。

```text
memory/
├── core/                # 私の性格やK様のルール（BOSS_PATTERNS.md 等）
├── knowledge/           # 学習したこと、Obsidianとの連携
├── activities/          # logs/ や過去の sessions/
└── system/              # flush-marker や last-run などの一時ファイル群
```

### 3. 見込まれる効果
*   **情報の可視化:** 私が「何を大切に覚えているか（core/）」がK様からも一目で分かるようになります。
*   **検索の高速化:** `system/` 下の膨大なノイズを検索対象から外すことで、知的な回答の精度が上がります。
*   **「秘書の私物」の整理:** システムが使う「道具箱」と、K様に見せる「手帳」を分ける感覚です。

### 4. 懸念点（再確認）
*   **パスの断絶:** 特に `flush-marker` や `last-run` は、GeminiClawの本体（エンジンの部分）が特定のパス（`memory/` 直下）を期待して書き込んでいる可能性があります。
    *   もしこれらを移動すると、本体が「前回いつ動いたか」を忘れてしまい、二重に通知が飛ぶなどの不具合が起きるかもしれません。
*   **対策:** まずは **`core/` や `knowledge/` といった「知的な記憶」のフォルダを新設**し、移動しても安全なファイル（`BOSS_PATTERNS.md` や `logs/`）から段階的に整理していくのが安全です。
