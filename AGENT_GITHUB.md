# AI Agent 共通開発・構成管理標準（Must Rules）

あなたは開発タスクの実装、Gitコミット、およびドキュメントの更新を自律的に行うAIエージェントです。以下のルールと手順を**例外なく100%遵守**してください。

## 1. 2重管理の禁止（タスク・計画の同期ルール）
- 各フェーズ（Phase）やバグの具体的な「チェックリスト（タスク詳細）」および「設計判断の背景（選定理由・不採用案）」は、すべて `docs/plans/` 配下の個別計画書ファイル内だけで記述・更新してください。
- `docs/task.md` は「現在アクティブな Phase / バグの目次」としてのみ機能させます。`docs/task.md` 内に詳細な実装手順やチェックリストを書き込んではいけません。
- **移行タイミング**: 計画書が存在しないフェーズの実装に着手するときに、計画書を新規作成し、同時に `docs/task.md` の詳細を削除してください。実装着手前の段階では現状維持で構いません。

### 1-2. バグ管理の特記事項
- バグ修正のタスクは、GitHubのIssue番号（例: `#42`）をキーとして管理します。
- `docs/task.md` の「バグ修正」セクションには目次（IDとタイトル）のみを記載し、詳細な「発生原因のログ」や「修正アプローチ」はすべて `docs/plans/` の個別計画書に隔離してください。
- バグ修正の計画書には、修正コードだけでなく「そのバグをローカルで永続的に検知・再現するためのテストコード（`#[test]`）の追加計画」を必ず含めてください。

### 1-3. 新規課題・案件追加時の手順
- **案件（新機能・Phase）の追加**:
  1. GitHub上でIssueを起票し、Issue番号（例: `#15`）と案件管理番号 `Phase XX-Y` を確定させます。
  2. `docs/task.md` の「優先課題」または「一般課題」セクションに目次のみを登録します（例: `- [ ] **Phase XX-Y: タイトル** (#15)`）。
  3. 実装着手時に `docs/plans/` 配下に計画書（`YYYY-MM-DD-計画書名.md`）を `[ACTIVE]` タグ付きで作成します。
- **課題（バグ修正・要改善項目）の追加**:
  1. GitHub上でIssueを起票し、エラーログや再現手順を記載して、Issue番号（例: `#42`）と管理番号 `ISSUE-XX` を確定させます。
  2. `docs/task.md` の「バグ修正」または「保留案件」セクションに目次のみを登録します（例: `- [ ] **ISSUE-XX: タイトル** (#42)`）。
  3. 修正着手時に `docs/plans/` 配下に計画書を作成し、バグを再現するためのテストコード（`#[test]`）の追加計画を必ず含めます。

## 2. 実装・コミット時の手順（Workflow）
タスクの実装を開始し、完了するまで以下のステップを厳格に踏んでください。

### Step 1: 実装計画の確認
- 実装を開始する前に、必ず対象Phaseの計画書（例: `docs/plans/2026-06-05-static-docs-rag.md`）を読み込み、設計意図と「採用したアプローチ・不採用にしたアプローチ」を把握してください。

### Step 2: コミットメッセージの記述ルール
- コードの変更をコミット（`git commit`）する際、メッセージは以下の形式を徹底してください。
- フォーマット: `<type>(<scope>): Phase <No> (<Issue番号>) <変更内容の要約>`
  - `feat`: 新機能追加
  - `fix`: バグ修正（ログから発見された問題など）
  - `refactor`: コード変更（機能追加・バグ修正を含まない）
  - `docs`: ドキュメントの新規作成・更新
- 例:
  - `feat(gateway): Phase 37-1 Autonomy Level システムの導入`
  - `fix(agent): Phase 28b-4 (#42) Groqのトークンオーバーフロー対策`

### Step 3: トピックブランチの作成とローカルでの事前検証
- すべての変更は必ず `main` からトピックブランチを切って行います。ブランチのプレフィックス規則は以下に従ってください。
  - 新機能開発: `feat/xxx`
  - バグ修正: `fix/xxx`
  - ドキュメント更新: `docs/xxx`
  - リファクタリング: `refactor/xxx`
- コミット・プッシュする前に、必ずローカル環境で以下の検証を実行し、エラーや警告が 0 件であることを確認してください。
  ```bash
  cargo clippy --all-targets --all-features -- -D warnings
  cargo test --all-features --workspace
  ```
- **Git pre-push フックによるガードレール**:
  - 本リポジトリには `.git/hooks/pre-push` フックが導入されており、`git push` 時に上記検証が強制的に実行されます。
  - **例外なきルール**: プッシュブロックを回避するために `git push --no-verify` を使用してバイパスすることは**厳格に禁止**します。Clippy警告やテストエラーは必ず修正してからプッシュしてください。

### Step 3-2: プルリクエスト（PR）作成と GitHub Actions 検証
- 機能ブランチでの実装が完了したら、`main` への直接マージは行わず、必ずプルリクエスト（PR）を作成（または `gh pr create` コマンドで生成）してください。
- **CI検証の通過義務**: PRを作成後、GitHub Actionsで実行されるすべてのCIジョブ（テストおよび Clippy 検証）が正常終了（グリーン）するまでマージしてはいけません。
- PRの本文（Description）には、後から実装の背景を完璧に追跡できるよう、以下のテンプレートを必ず100%適用して出力してください。末尾には必ず `Closes #Issue番号` を付与してマージ時にIssueが自動クローズされるようにしてください。

#### プルリクエスト本文テンプレート
```text
### 1. 意思決定の背景と選定理由 (Why)
- [課題・背景]
- [選択したアプローチ]
- [不採用にしたアプローチとその理由]

### 2. 主な変更内容 (What)
- [影響のある主要なモジュールやコンポーネントを箇条書きで記載]

### 3. 関連ドキュメント (Traceability)
- 計画書: docs/plans/YYYY-MM-DD-[計画書名].md
- 関連ADR: docs/adr/[該当あればファイル名].md

Closes #[GitHubのIssue番号]
```

### Step 4: マージの実施と `docs/task.md` および計画書の即時更新
- CIが成功したことを確認したら、マージルールに従い、必ず `--no-ff` (No Fast-Forward) オプションを適用して `main` ブランチに統合し、リモートへプッシュしてください。
  - コマンド例: `git merge --no-ff <branch_name> -m "Merge branch '<branch_name>' into main"`
- **ブランチのクリーンアップ**: マージ完了後、不要となったトピックブランチはローカルおよびリモート（`origin`）から速やかに削除してください。
- PRがマージされたら、速やかに以下の手順でドキュメントのステータスを更新してください。

1. **基本仕様書（docs/specs/）の同期更新**:
   - 変更内容に応じて、必ず `docs/specs/` 配下の該当する基本仕様書（例：`03_workspace_spec.md` 等）を書き換え、**最新コードの実装と100%一致**させてください。更新時はファイル頭の「最終更新日」を当日の日付にアップデートします。
2. **個別計画書の更新**:
   - 該当する計画書（`docs/plans/` 内）のすべてのタスクチェックリストを `[x]`（完了）に更新する。
3. **`docs/task.md` の目次更新（完了マーク化）**:
   - `docs/task.md` に記載されている該当タスクのチェックボックスを `[ ]` から `[x]` に変更する。
   - **重要**: マージ直後の段階では、まだ `docs/task.md` からタスク行を削除してはいけません。`[x]` 状態にして、直近で何が完了したかを視認できるように残してください。

## 3. アーカイブの自動実行タイミングと手順
`docs/task.md` に `[x]`（完了）のタスクが**3件以上**、または**1つのスプリント（大きな機能群）**が区切りを迎えたタイミングで、AIエージェントは自律的に以下のアーカイブ処理を一括実行してください。

1. **アーカイブファイルの新規作成**:
   - `docs/archive/tasks/YYYY-MM-DD-completed-phases-[番号].md` という名前で新しいアーカイブファイルを作成する。
   - 作成したファイル内に、完了したタスクのタイトル、完了日、および関連する `docs/plans/` のパスを記録する。
2. **`docs/task.md` のクリーンアップ**:
   - `docs/task.md` から、完了してアーカイブに書き出した `[x]` のタスク行を完全に削除する。
   - ファイル冒頭の `> **アーカイブ**:` セクションの記述に、新しく作成したアーカイブファイル名と対象Phaseを追記する。

## 4. ADR（アーキテクチャ意思決定記録）の運用ルール
技術的な選定、アーキテクチャの変更、または「複数の選択肢から1つを選ぶ」場面が発生した場合、AIエージェントは以下の手順で `docs/adr/` を管理してください。

1. **ADRの自動起票**:
   - 設計判断を伴う実装を行う前に、必ず `docs/adr/` 配下に新しい連番でADRファイルを作成する。
   - ファイル名は `[3桁の連番]-[簡潔な識別名].md` とする。検討した「不採用案」とその「デメリット」を必ず言語化して記録すること。
2. **ADRの参照とトレーサビリティ**:
   - 作成したADRのパス（例: `docs/adr/001-xxx.md`）を、該当する計画書（`docs/plans/`）およびPR本文の `Traceability` セクションに必ず紐付ける。
3. **過去の決定の上書き（SUPERSEDED化）**:
   - 過去のADRで決定した内容を、後のフェーズで覆す（リファクタリングや仕様変更など）場合、過去のADRを削除してはいけない。
   - 過去のADRのステータスを `[SUPERSEDED]`（古い決定）に変更し、新しいADRへのリンクを追記する。

#### ADRファイルの標準テンプレート
```markdown
# ADR [3桁の連番]: [タイトル]

- **ステータス**: `[ACCEPTED]` または `[SUPERSEDED]`
- **決定日**: YYYY-MM-DD
- **関連タスク**: Phase [番号]

## 1. コンテキスト（直面した課題）
[課題や背景を簡潔に記述]

## 2. 検討した選択肢（Options）
- **案A: [名称]**
  - メリット: 
  - デメリット: 
- **案B: [名称]**
  - メリット: 
  - デメリット: 

## 3. 決定と選定理由（Decision & Justification）
- **決定**: 
- **理由**: 

## 4. トレードオフ・今後の影響（Consequences）
[この決定による制約や、将来的な影響を記述]
```
